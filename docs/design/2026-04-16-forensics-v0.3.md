## Why this is the announcement target

Half-baked first releases followed by months of progression don't get re-evaluated. Launch one: launch loud, with a complete story, and the right lightbulbs turn on. v0.1 (local dev secrets) and v0.2 (Dynamic Secrets foundations) ship quietly. **v0.3 is the announcement** — the one where the story holds up to scrutiny and has real substance behind every claim.

The story v0.3 tells:

> AgentKMS is the secret-issuing authority for AI agents. When a credential leaks — found in git history, reported by a security researcher, detected by a honeytoken — you get a complete forensic report in under a minute: who issued it, who used it, why, at what scope, and for how long.

Not just vending. Not just attribution. **Chain of custody.**

## The two lenses on the same data

Same audit substrate; two buyer personas.

### Engineering manager — observability / anomalies

- Per-user rolling baselines: tokens issued per day, provider spend, model distribution, time-of-day histogram
- Deviation alerts: "Frank issued 47 GitHub PATs in the last hour (baseline: 3/day)"
- Coordination patterns: "6 developers asked for `prod-db-readonly` within 10 minutes — expected wave or coordinated anomaly?"
- Session-level oddities: "This Cursor session requested 4 credential classes; p95 is 2"

Dashboard UX. Trend lines. Aggregates.

### Security engineer — incident response

Same data, different query. "GitHub secret scanning reported `ghp_ABCxyz` at 10:47am. Tell me everything."

Single-credential query. Full lifecycle. Correlation to upstream events. Executive-readable summary.

## The forensics report, concretely

Targets: **under 30 seconds end-to-end**, one-screen terminal output, every line lands. The money shot is "credential expired 63h before leak detection" — dynamic secrets plus forensic chain-of-custody equals leaks that don't matter.

```bash
$ akms forensics inspect ghp_ABCxyz

  Leaked credential found in audit ledger.  ✗ Blast radius: BOUNDED.

  ──────────────────────────────────────────────────────────────────
  Scope:         acmecorp/legacy-tool
                 (1 repo; contents:write + pull_requests:write)
  TTL applied:   8h
  Lifecycle:     issued  2026-04-13 15:20 UTC  (to frank@acmecorp)
                 expired 2026-04-13 23:20 UTC  ← credential already dead
                 leaked  2026-04-16 10:47 UTC  ← reported 63h after expiry

  Usage during live window (8 hours):
    15:22  clone acmecorp/legacy-tool       ✓ expected
    15:47  push branch migration-v3          ✓ expected
    16:31  open PR #47                       ✓ expected

  Post-expiry usage: NONE (credential was dead when leaked)

  ──────────────────────────────────────────────────────────────────
  ✓ Assessment: no damage — credential expired 63h before leak detection.
  ✓ Action required: none. GitHub has already revoked the dead token.

  Full issuance context:
    Requester:   frank@acmecorp  (CN=frank, O=acmecorp, OU=platform-team)
    Session:     claude-code-session-8f3a
    Via:         MCP tool `vend_github_pat`
    Policy:      allow-github-for-developers (rule #4)

  ℹ Single-credential inspect only on OSS. Install c9-forensics-plus
    for correlated multi-credential queries and org-wide search.
```

That report answers every forensic question a security engineer asks — **who issued, who used, why, scope, exposure window** — and its structure foregrounds the decision the engineer has to make (act / don't act). The blast-radius assessment at top, dead-credential timing in the lifecycle block, and explicit "Action required: none" close the ticket in under a minute.

## Data substrate required

See [audit-schema-migration](2026-04-16-audit-schema-migration.md) for the detailed gap analysis. Summary:

- **Bucket A (v0.1.1, now):** additive audit fields — UUID, rule ID, cert serial, token hash, credential type, caller OU.
- **Bucket B (v0.2):** scoped credential vending (so scope-at-issuance is real, not a placeholder) + revocation event types + server-side usage tracking.
- **Bucket C (v0.3):** post-hoc detection enrichment + upstream log ingestion + forensics CLI + query UX.

## Honeytokens — v0.3 stretch or v0.4

AgentKMS holds provider admin credentials (see dynamic-secrets doc). It can also mint **honeytokens**: credentials that look valid but fire an alert on any use. Plant in:

- Public `.env.example` files
- Test fixtures checked into repos
- Staging environment variables
- Slack DMs to self

Any use is proof of scraping. Usage logs tell you from where. Free intelligence on attackers, generated by your own infrastructure.

v0.3 if we can fit it, v0.4 otherwise. The leak-webhook infrastructure v0.3 builds is most of what's needed.

## What this is not

- **Not** a replacement for SIEM. AgentKMS is authoritative about credential lifecycle; SIEM is authoritative about everything else. Export stream, don't try to be the inbox.
- **Not** a secret scanner. We don't scan git repos for leaks — we receive leak reports from GitHub/others and correlate. Other tools do the scanning better.
- **Not** an anomaly-detection AI. Baseline + rolling average + thresholded alerts. Good enough for 95% of cases. ML-driven anomaly scoring is a future offering, not v0.3.

## Open questions

- **Self-hosted vs. hosted ingestion.** The ingestion workers need credentials to the upstream admin APIs. In self-hosted deployments those live in the customer's infra, which is fine. Hosted ingestion ("let Catalyst9 ingest on your behalf") would require customer upstream credentials to live in our infra — rejected by the sovereignty principle. Self-hosted only.
